1. Installation

1.1. Hardware Requirements

BAli-Phy should be able to run on any hardware with a modern operating system, including Linux or UNIX, Windows, and Mac OS X.

We recommend at least 512Mb of RAM and a 1.5GHz processor. We typically run BAli-Phy on Pentium 4 processors with 1Gb of RAM and a 3 GHz processor.

1.2. Operating System requirements

1.2.1. Linux and UNIX

A Linux or UNIX system should be able to install and run BAli-Phy without any modification.

1.2.2. Windows

On Windows you must first install cygwin. Cygwin is a Linux environment for Windows, and includes GCC, GNU make, and other software. You can access the Cygwin command line (not the normal windows command line) through the Start menu.

1.2.3. Mac OS X

A Mac OS X system is based on UNIX, and should be able to compile and run BAli-Phy. The version of OS X must be 10.1 or higher. If you want to use the development tools from Apple to compile BAli-Phy, the system must be 10.4 or higher, and you must upgrade to XCode 2.1 (see Software Requirements).

1.3. Software Requirements

1.3.1. Compilation Requirements

The following software packages are required for compiling BAli-Phy:

  • A modern C++ compiler (GCC version 3.4 and 4.0 are known to work)

  • GNU make version 3.80 or higher (GNU make).

  • The GNU Scientific Library (GSL)

  • The BOOST C++ Library (BOOST) version 1.32 or higher.

Mac OS X issues:

If you want to use the development tools shipped by Apple (e.g. GCC) then you need OS X version 10.4 (Tiger) or higher, and you need to install XCode 2.1. Otherwise you can compile GCC yourself from the FSF sources.

Fink provides Linux software for Mac OS X. You may use it to install the GSL library and include files, but the BOOST libraries that come with fink will not work, because they are compiled with GCC 3.3. You must therefore compile the BOOST libraries yourself, using GCC 3.4 or higher.

1.3.2. Analysis and Documentation Requirements

In addition, the following software packages are recommended, although not required:

  • The plotting program gnuplot (gnuplot)

  • GNU make version 3.80 or higher (GNU make).

  • Mozilla or Mozilla/Firefox to view the math in the XHTML documention. (Firefox)

1.4. Compiling BAli-Phy

In order to compile the program on UNIX, simply enter the src/ directory and type make install at the shell prompt. (This requires GNU make.) This will compile bali-phy and its associated tools, and copy them to the directory ../build/bin.

1.4.1. Selecting a non-default C++ compiler

The default C++ compiler is g++. On some systems, g++ invokes GCC version 3.3, and the correct compiler is called something else, such as g++-4.0. To use g++-4.0 as the C++ compiler when compiling BAli-Phy, you can set the CXX environment variable as follows: 

% CXX=g++-4.0 make install

To change this, or to change other compilation options, edit the makefile src/GNUmakefile.

1.4.2. Optimizing for a specific architecture

On ix86 computers, you can add either ARCH=pentium3, ARCH=pentium4, or ARCH=athlon-xp to the beginning of the command line. This allows the compiler to generate faster code that only works on the specified architecture.

1.4.3. Statically linked executables

You can also add STATIC=y in order to build a static executable.

1.5. Installing the executables

In order to use bali-phy and the tools that you have just compiled, you must copy them into a directory in your path. We recommend copying all tools in build/bin/ into the directory ~/bin/ and adding this directory to your path.

In addition, it is recommended that you make a ~/bin/bali-phy/ directory, and copy the bali-phy executable to the ~/bin/bali-phy/version. This will allow the ./bin/bali-phy.sh script to find your executable for use with the SGE (Sun Grid Engine) environment.

2. Running the program

2.1. The simple way

The simplest way to run BAli-PHy is to invoke the bali-phy executable directly: 

% ./bali-phy sequence-file >out 2>err

Here sequence-file is a FASTA or PHYLIP file containing the sequences you wish to analyze.

Note

In many cases it is necessary to specify the location of the data directory in order to run the program. The data directory defaults to ./Data if not explicitly set. (See option --data-dir).

2.2. The recommended way

The recommended way to run the program is to copy the bali-phy executable to ~/bin/bali-phy/version and run (in a bourne-shell) 

% export BALI_PHY_DATA=data-dir
% VERSION=version CLASS=class bali-phy.sh sequence-file [OPTIONS]

The wrapper script bali-phy.sh will setup and run bali-phy for you. This involves:

  1. Creating a unique directory in which to run each invocation of BAli-Phy. (The directories will be named class-1, class-2, etc.)

  2. Linking the BAli-Phy data directory into the run directory for you.

  3. Executing BAli-Phy inside the run directory, using the SGE scheduling engine if present.

This setup is important because it ensures that multiple invocations of BAli-Phy will produce output files in different directories, and will not over-write each other.

3. Command-line options

3.1. Usage

The syntax for the program is:

bali-phy {sequence-file} [OPTIONS]

The sequence file is the only required argument. It can be either a FASTA (*.fasta) file or a PHYLIP (*.phy) file. The file must end in one of these two suffixes or BAli-PHy won't know how to read it. In addition your FASTA files should not contain any blank lines.

The optional arguments are described below.

3.2. General options

--help

Show help message.

--version

Show version information.

--show-only

Analyze initial values and exit.

--data-dir directory

Specify the data directory.

--align-constraint file

Specify a file with alignment constraints

--internal +

Set all internal node entries wildcards initially.

--gaps star

Don't use indel info, and fix the alignment.

--letters star

Use a star tree for the substitution model.

3.3. MCMC options

--iterations number=100000

Specify the number of iterations to run.

--subsample factor=1

Specify a factor by which to subsample.

--T temperature=1

Specify a temperature for MCMCMC.

--enable move

Enable a comma-separated list of transition kernels.

--disable move

Disable a comma-separated list of transition kernels.

3.4. Parameter options

--randomize-alignment

Randomly re-align sequences before use.

--tree file

Specify file with initial tree.

--set parameter=value

Specify initial value of parameter.

--fix parameter[=value]

Mark parameter fixed, and optionally specify a value.

--unfix parameter[=value]

Mark parameter not fixed, and optionally specify an initial value.

--frequencies frequencies

Specify initial frequencies: 'uniform','nucleotides', or a comma-separated list of frequencies.

3.5. Model options

--alphabet name

Specify the alphabet: DNA, RNA, Amino Acids, Amino Acids + stop, Triplets, Codons, or Codons + stop.

--smodel name

Specify the substitution model.

--imodel name

Specify the indel model.

4. Examples

Here are some examples which demonstrate how to run BAli-PHy:

Example 1. No frills

Here we analyze the EF-Tu 5-taxon data set provided with the software. The alignment is randomized before the chain begins.

 % bali-phy Data/EF-Tu/5d.fasta --randomize_alignment >out 2>err

Example 2. Multiple-Rate Substitution Model

We now modify the previous example by changing the substitution model to allow gamma-distributed rate variation and invariant sites. The amount of rate variation and the fraction of invariant sites are estimated

 % bali-phy Data/EF-Tu/5d.fasta --smodel gamma+INV --randomize-alignment >out 2>err

Example 3. Fixed alignment

Here we use the 5S rRNA 5-taxon data set provided with the software. The alignment is fixed and the traditional likelihood model is used, making indels non-informative. In addition, the transition kernel which samples nucleotide frequencies is disabled, fixing the nucleotide frequencies to empirical values estimated from the input sequences. 

 % bali-phy Data/5S-rRNA/5d.fasta ---disable frequencies --gaps star >out 2>err

5. Input

5.1. Sequence formats

BAli-PHy can read in sequences and alignments in both FASTA and PHYLIP formats. However, in FASTA format, the file must not contain blank lines between the sequences, because BAli-PHy will stop reading in sequences when it sees a blank line.

5.2. Is my data set too large?

Large data sets run more slowly than small data sets. We recommend a conservative starting point with few taxa and short sequence lengths. You can then increase the size of your data set until a balance between speed and size is reached.

The number of samples that you need depends on whether you are primarily interested in obtaining a point estimate or in obtaining detailed measures of confidence and uncertainty. For detailed measures of confidence and uncertainty you should obtain a minimum of 10,000 samples after the Markov chain converges. For an estimate, you don't need very many samples after convergence. (But you may need many samples to be sure that you've converged!)

5.2.1. Too many taxa?

BAli-PHy is quite CPU intensive, and so we recommend using 12 or fewer taxa in order to limit the time required to accumulate enough MCMC samples. We note that many phylogenetic hypotheses can be reduced to hypotheses concerning only four taxa. We recommend removing as many taxa as possible from your data set. You can then add taxa back into the data set one you have determine how much this will increase the duration of the run.

5.2.2. Sequences too long?

Aligning sequences takes O(L 2 ) time and RAM, where L represents the sequence length. Thus, doubling the length of the input sequences increases RAM requirements and run time by a factor of 4. Therefore sequence lengths beyond 750 letters may prove impractical due to limited RAM or the speed of the calculation. For protein sequences, we recommend coding the sequences in terms of amino acids or codons, rather than nucleotides.

However, you might try to see how long you can make your sequences before the attempt becomes impractical.

6. Models

6.1. Substitution models

All of BAli-PHy's substitution models are mixtures of continuous-time Markov chains (CTMC). These mixtures can consist of the same CTMC model run at a mixture of rates (e.g. a gamma rate model) or parameters (e.g. a Yang M2 model), or even a mixture of different models.

6.1.1. Default substitution models

If the substitution model is not specified, then the default model for the alphabet is used. For DNA or RNA, the default model is HKY. For Codons, the default model is YangM0. For AminoAcids, the default model is Empirical[WAG].

6.1.2. Specifying a substitution model

Substitution models are specified using a stack, as follows: Model[arg]+Model[arg]+...+Model[arg] where each model uses the previous models as input. Arguments are optional.

The basic CTMC models are EQU, HKY, TN, Empirical, and YangM0.

Model modifiers are gamma, INV, YangM2, YangM3, and YangM7.

6.1.3. Examples

Example: --smodel EQU --alphabet Triplets

Example: --smodel HKY

Example: --smodel Empirical[WAG]+gamma+INV

Example: --smodel YangM0 --alphabet Codons

Example: --smodel YangM2 --alphabet Codons

Example: --smodel YangM0[TN]+YangM2 --alphabet Codons

6.1.4. Extended model descriptions

All basic CTMC models are reversible, and can be characterized by rates

Q ij=S ij×π j fπ i 1 -f,

where S is a symmetric matrix and π denotes the equilibrium frequencies for the Markov chain. The transition matrix for a time t is then given by

P(t) ij=e Q ij×t,

using a matrix exponential.

6.2. Indel models

The current models are simple, fragments, and fragments+T. The default is fragments+T.

6.3. Alignment constraints

To pin columns of the alignment, specify alignment constraints in a file as follows:

  1. Use the argument --align-constraint filename

  2. The filename refers to a file in which each line represents a constraint.

The first line of the file is a header consisting of an ordered list of sequence names separated by spaces. Each subsequent line consists of a space-separated list of sequence positions, with the first position corresponding to the first leaf sequence, the second position corresponding to the second leaf sequence, etc. Thus, if there are <n> leaf taxa, then each line corresponds to a space-separated list of <n> integers.

For example, the line

1 2 2

means that position 1 of leaf sequence 1 is aligned to position 2 of leaf sequences 2 and 3. Note that the first position in a sequence is position 0.

Optionally, one may use a '-' instead of an integer, which denotes a lack of constraint for that sequence. This can be useful as follows:

2 2 - - 
- - 2 2

The above constraints force alignment between position 2 of sequences 1 and 2, and between position 2 of sequence 3 and 4.

7. Output

7.1. Output streams

BAli-PHy writes out the sampled alignments and other information to the standard output stream, which displays on the terminal by default. Error messages are written to the standard error stream, which also displays on the terminal by default. In addition some other useful information is written to the standard error stream, such as the acceptance fraction for the various transition kernels.

You should redirect the standard output to a file called out and redirect the standard error to a file called err so that the information on these streams is available for later analysis. In addition, some scripts assume that the names out and err are used. The way to redirect these streams to the files is to specify >out 2>err on the command line.

7.2. Output files

When run in a clean directory, BAli-PHy produces the following output files. Except for the file MAP which contains alignments, each line in these files should correspond to one iteration.

Pr

prior, likelihood, probability

pI

indel model parameters

pS

substitution model parameters

trees

tree samples

MAP

successive estimates of the MAP point

8. Analyzing the output

8.1. Running the analysis

Copy (or link, using ln -s) the file bin/GNUmakefile into the current directory. Then run the command: 

 % SKIP=burn-in make

or 

 % SKIP=burn-in SIZE=iterations make

Here, burn-in represents the number of iterations to skip as burn-in, and iterations represents the maximum number of iterations to use in the analysis. In order for this to work, you must have having built the tools in the tools/ directory and have placed them in your path.

Running the analysis will produce the three directories: Results/, Mixing, and Work/. The Results/ directory contains tree and alignment estimates and confidence levels. The Mixing/ directory contains various measures of whether the chain ran long enough given its Mixing. The Work/ directory contains various intermediate files that you probably do not need.

8.2. Results

Running the above command will produce the following files in the Results/ directory:

analysis

A description of the analysis: burn-in, length, time completed.

Report

A summary of results: posterior means and credible intervals, etc..

consensus

Summary of MAP and consensus topologies, as well as supported partitions.

c-levels.plot

The number of partitions supported at each LOD level.

clevel.topology

The consensus topology at level level (Newick format)

c50.tree

The majority consensus topology + branch lengths (Newick format)

MAP.topology

An estimate of the MAP tree (Newick format)

MAP.tree

An estimate of the MAP topology + branch lengths (Newick format)

MAP.fasta

In addition, if you type SKIP=burn-in make Results/MAP-AU.html, the following files will be produced:

MAP-AU.html

An AU plot

MAP-AU.prob

The probabilities for each letter in the AU plot

8.3. Mixing

In addition, the following files will be produced in the Mixing/ directory:

partitions.bs

Confidence intervals on the support for partitions, generated using a block bootstrap.

SRQ.plot

A collection of SRQ plots for the supported partitions.

sum.plot

A collection of sum plots for the supported partitions.

sumi.plot

A collection of inverse sum plots for the supported partitions.

Plots can be viewed by typing plot 'file' with lines in gnuplot.

9. Auxiliary tools

9.1. trees-consensus

Usage: tree-dist-compare file1 [file2 ... ] [OPTIONS]

This program analyzes the tree sample contained in file1. It reports the MAP topology, the supported taxa partitions (including partial partitions), and the majority consensus topology.

9.2. trees-bootstrap

Usage: trees-bootstrap file1 [file2 ... ] --predicates predicate-file [OPTIONS]

This program analyzes the tree samples contained in file1, file2, etc. It gives the support of each tree sample for each predicate in predicate-file, and reports a confidence interval based on the block bootstrap.

Each predicate is the intersection of a set of partitions, and is specified as a list of partitions or (multifurcating) trees, one per line. Predicates are separated by blank lines.

9.3. trees-to-SRQ

Usage: trees-to-SRQ predicate-file [OPTIONS] trees-file

This program analyzes the tree samples contained in trees-file. It uses them to produce an SRQ plot for each predicate in predicate-file. Plots are produced in gnuplot format, with one point per line and with plots separated by a blank line.

If --mode sum is specified, then a "sum" plot is produced instead of an SRQ plot. In this plot, the slope of the curve corresponds to the posterior probability of the event. If the --invert option is used then the slope of the curve correspond to the probability of the inverse event. This is recommended if the probability of the event is near 1.0, because the sum plot does not distinguish variation in probabilities near 1.0 well.
